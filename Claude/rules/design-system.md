# 🎨 Design System

-----

## Philosophy & Aesthetic Direction

The target aesthetic is **modern OS / precision tool** — think Linear, Vercel, Raycast, Arc,
macOS Sequoia. Interfaces that feel like they were designed by a small team who cared
obsessively about craft, not generated by an AI that defaulted to "clean and modern."

The defining qualities of this aesthetic:

- **Restraint is the skill.** Every element earns its place. Decoration that doesn't
  carry meaning is noise. When in doubt, remove it.
- **Density is not clutter.** Information-dense UIs can feel calm when spatial rhythm
  is precise. Generous whitespace and precise whitespace are both valid — arbitrary
  whitespace is neither.
- **Hierarchy through weight and spacing, not color.** Color is reserved for state
  (error, success, active) and a single accent. Using color to create hierarchy is
  a sign the layout isn't working.
- **Surfaces behave like materials.** Subtle differentiation — slightly lighter border,
  1px inset shadow, 2% opacity shift — creates depth without fake gradients.
- **Motion is mechanical, not decorative.** Transitions communicate state change.
  They don't "delight" — they orient. Keep them short, physical, purposeful.

-----

## ❌ Prohibited Patterns (AI Tells)

These are the patterns that make a site look AI-generated. Do not use them.

### Visual
- **Emoji as icons or decoration** — use SVG icons or Unicode symbols with intention.
  Emoji in body copy or UI elements reads as chatbot, not product.
- **Purple/blue/teal hero gradients** — the single most identifiable AI aesthetic tell.
- **Gradient text** (`background-clip: text`) — overused to the point of meaninglessness.
- **Glassmorphism on everything** — `backdrop-filter: blur()` is a detail, not a motif.
  One frosted surface per layout, maximum.
- **Blob/orb background shapes** — decorative SVG blobs signal "I ran out of ideas."
- **Multi-color CTA gradients** — buttons are affordances, not billboards.
- **Border-radius over 12px on rectangular containers** — cards are not pills.
  `--radius-xl` is for modals and sheets only; `--radius-md` is the default card radius.
- **Shadow stacking** — one shadow per element. Stacking `box-shadow` values to fake
  "material depth" looks cheap. Use the shadow scale.
- **Decorative dividers with gradient fades** — `border-top: 1px solid var(--color-border)`
  is almost always the right call.

### Layout
- **Three-column "feature cards" with emoji icons** — the most recognizable AI layout pattern.
  If you're building a marketing section, vary the structure: asymmetric, editorial, staggered.
- **"Stats" sections** ("10K+ users", "99.9% uptime") — generic filler. Replace with
  something specific to the actual project or cut it.
- **Hero → Features → CTA → Footer** — this exact sequence is AI default output.
  Interrupt it. Reorder. Merge sections. Make structural choices.
- **Identical card padding everywhere** — vary container density intentionally.

### Copy & Content
- **Placeholder emoji headers** in section titles (🚀 Our Features, ✨ Why Choose Us)
- **"Powered by AI" badge** unless the project is specifically about AI
- **Lorem ipsum** — always use real or realistic copy, even in scaffolds

### Animation
- **Entrance animations on every element** — stagger is a seasoning, not a base coat.
  Animate the most important 1–2 elements on page load, nothing else.
- **Hover animations that scale cards up** (`transform: scale(1.05)`) — it's 2019.
  Use border color shift, background lift (`translateY(-2px)`), or shadow change instead.
- **Transition: all** — always specify the property. `transition: all` is lazy and causes
  jank on properties that shouldn't animate (height, display, clip-path).

-----

## ✅ Patterns That Signal Craft

- **Optical spacing** — space around text elements should feel even optically, not
  measure identically. Headings often need less top-margin than the grid suggests.
- **Tabular numbers** (`font-variant-numeric: tabular-nums`) on any column of numbers.
- **Tight letter-spacing on large type** (`letter-spacing: -0.02em` to `-0.04em` at 3xl+).
  Large type with default tracking looks unset.
- **ALL CAPS labels with wide tracking** (`letter-spacing: 0.1em`) for metadata, eyebrows,
  and category tags — signals intentionality.
- **1px borders that use opacity** (`rgba(255,255,255,0.08)`) rather than a flat color —
  they read as part of the surface material, not drawn on top of it.
- **Focus states that feel native** — match the OS focus ring style (offset outline,
  not a glow shadow). Don't skip them; make them beautiful.
- **Empty states with personality** — a well-designed empty state is a trust signal.
  It means someone thought about the full lifecycle.
- **Consistent interactive affordance language** — if hover is `--color-surface-2`, that
  means the same thing everywhere. Don't mix hover paradigms.
- **Monospace for metadata** (`--font-mono`) — timestamps, version numbers, IDs, file sizes.
  It signals precision.

-----

## Interaction States (Required for All Interactive Elements)

Every interactive element must have all four states defined. Unstyled states are bugs.

| State    | Default pattern |
|----------|----------------|
| Default  | Base styles |
| Hover    | `background: var(--color-surface-2)` or `translateY(-2px)` + shadow lift |
| Active   | `translateY(0)` + slight brightness reduction (feels physical) |
| Focus    | `outline: 2px solid var(--color-accent); outline-offset: 3px` |
| Disabled | `opacity: 0.4; cursor: not-allowed` — never just `color: var(--color-muted)` |

Transition spec for interactive elements:
```css
transition: background var(--duration-fast) var(--ease-out),
            transform  var(--duration-fast) var(--ease-out),
            box-shadow var(--duration-fast) var(--ease-out),
            border-color var(--duration-fast) var(--ease-out);
```
Never use `transition: all`.

-----

## Design Decision Checklist

Ask these before committing any visual implementation:

1. **Does every element serve the hierarchy?** If you can't explain why it's there, remove it.
2. **Is color being used for state or decoration?** If decoration, remove it.
3. **Does this pattern exist in macOS, Linear, or Raycast?** If not, justify the deviation.
4. **Are there emoji in the UI?** Remove them unless the project brief specifically calls for them.
5. **Is there a gradient?** If it's not a subtle surface gradient (max 5% lightness shift), justify it.
6. **Are all four interaction states defined?** If not, finish them before moving on.
7. **Does the motion communicate something?** If it's purely decorative, cut it.

## Token Reference

> NEVER invent new token names mid-session. If a value isn't here, add it to this
> file first, then use it. This prevents token drift across sessions.

### Color

```css
/* Surface hierarchy */
--color-bg:        #0f0f0f;   /* Page background */
--color-surface:   #1a1a1a;   /* Card, panel, raised surface */
--color-surface-2: #242424;   /* Nested surface, hover state */
--color-border:    #2e2e2e;   /* Dividers, outlines */

/* Text hierarchy */
--color-text:      #ededed;   /* Primary body copy */
--color-muted:     #999999;   /* Secondary, captions, placeholders */
--color-subtle:    #555555;   /* Disabled, decorative */

/* Accent & feedback */
--color-accent:    #e8632a;   /* CTAs, highlights, focus rings */
--color-accent-2:  #c8502a;   /* Hover state on accent */
--color-success:   #4caf7d;
--color-warning:   #f0a500;
--color-error:     #ff5555;
```

> **Override here** when a project deviates — never in one-off selectors.

### Token Override Rules

- **New project-specific tokens** → add to `frontend/style.css` `:root` and document here.

- **Never** use a hardcoded hex, px, or `cubic-bezier` if a token exists.
